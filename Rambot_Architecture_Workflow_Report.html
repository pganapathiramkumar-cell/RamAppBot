Solution Shape

Client Layer

Two frontends exist: apps/web and apps/mobile.

  • Web uses App Router pages for Steer, Skill, and Documents.
  • Mobile uses Expo Router with tab navigation.
Domain Services

The backend is split into API Gateway, Steer, Skill, Auth, AI Orchestrator, Notification, and Document Service.

  • Steer and Skill follow presentation, application, domain, infrastructure layers.
  • Document Service is pipeline-oriented for async PDF analysis.
Platform Layer

Infrastructure centers on PostgreSQL, Redis, RabbitMQ, Docker Compose, and optional Kubernetes.

  • Shared TS packages expose API client and types.
  • AI providers are configurable across local and cloud runtimes.

Port 8000 API Gateway

FastAPI gateway with auth middleware, rate limiting, logging, gzip, CORS, and reverse proxy routes.

Port 8001 Steer Service

Strategic goal management with async SQLAlchemy, Redis cache, RabbitMQ publishing, and AI scoring support.

Port 8002 Skill Service

Skill catalog and lifecycle transitions backed by SQLAlchemy, Redis, RabbitMQ, and AI helper logic.

Port 8003 Auth Service

Auth API scaffold exists, but login/register/refresh/me currently return 501 placeholders.

Port 8004 AI Orchestrator

Dedicated FastAPI service for steer analysis and skill gap analysis via Ollama-backed agents.

Ports 8005 / 8006 Notification + Document

Notification consumes events; Document handles upload, validation, background analysis, and retrieval APIs.

Persistence, caching, and messaging sit behind the services.
Data PostgreSQL 16-alpine

Compose provisions steer, skill, and auth databases. Document code currently uses in-memory storage helpers.

Cache Redis 7-alpine

Used for rate limiting and service-local caching with separate DB indexes per service.

Messaging RabbitMQ 3.13-management-alpine

Event broker for publication/consumption. Some services fall back to in-memory publishers if unavailable.

AI Runtime Cloud + Local LLM

Groq, NVIDIA NIM, Cerebras, and Ollama are supported depending on service and environment settings.

Observed implementation nuance: the web document feature uses NEXT_PUBLIC_DOCUMENT_API_URL and calls document-service directly for uploads and analysis polling, while other feature areas use NEXT_PUBLIC_API_URL through the gateway.

Primary Workflows

1Steer Workflow

Clients send Steer requests to API Gateway, then the gateway proxies them to Steer service.

  • Gateway applies logging, auth, rate limiting, compression, and CORS.
  • Steer endpoints are fully wired with FastAPI dependency injection.
  • Startup initializes PostgreSQL tables, Redis, RabbitMQ, and an LLM client.
  • LLM fallback chain in Steer is Groq → NVIDIA NIM → Cerebras → mock.

2Skill Workflow

Skill traffic also enters through API Gateway and is handled by the Skill service lifecycle workflows.

  • Supported actions include create, list, search, submit-review, approve, deploy, metrics update, and deprecate.
  • Skill mirrors the same infrastructure pattern as Steer: PostgreSQL, Redis, RabbitMQ, and multi-provider AI fallback.

3Document Analysis Workflow

The document feature is the clearest end-to-end async workflow in the repo.

  • User uploads a PDF from the web app directly to document-service on port 8006.
  • Document Service validates file type, size, and empty input.
  • Upload returns document metadata and starts a FastAPI background task.
  • The background task extracts text and runs three AI chains in parallel: summary, entity extraction, and Mermaid workflow generation.
  • Frontend polls /api/v1/analyses/{id} every 3 seconds until status becomes done or failed.

4AI Orchestration Workflow

AI Orchestrator exposes separate endpoints for steer analysis and skill gap analysis.

  • It creates specialized agents and sends context to Ollama-backed analysis routines.
  • Current orchestrator settings default to OLLAMA_MODEL=llama3.2.
  • Document-service AI does not use the orchestrator; it uses its own provider fallback logic.

5Notifications and Events

Notification service is positioned as the event consumer layer.

  • RabbitMQ is part of the compose stack and notification-service depends on it.
  • Firebase push support is present through firebase-admin==6.5.0.
  • Steer and Skill publish events or fall back to in-memory publishers if the broker is unavailable.

Implementation Notes and Gaps

What Is Strong
  • Clear service decomposition by domain.
  • Steer and Skill use layered architecture consistently.
  • Frontend is split by features and aligned to domain areas.
  • Document-service has a concrete async processing pattern.
  • Multi-provider AI fallback is already implemented.
What Looks Transitional
  • Auth service endpoints are not yet wired to real use cases.
  • Document feature bypasses the gateway, so ingress is not fully uniform.
  • Document-service currently uses in-memory storage helpers in code.
  • Python base image versions are not fully standardized.
  • Some older architecture markdown is broader than the currently wired implementation.
This report reflects the current implementation in code, not only the intended target architecture. The repository contains both implemented flows and planned or stubbed capabilities.
