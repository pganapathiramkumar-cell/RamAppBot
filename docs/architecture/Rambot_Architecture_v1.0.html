1. System Architecture Overview

Rambot Enterprise is a Turborepo monorepo split into three layers: client apps (Web + Mobile), backend microservices (7 FastAPI services), and a shared data/AI tier. All services communicate through a central API Gateway. The AI tier uses Anthropic Claude Sonnet 4.6 for strategic analysis and a local RAG pipeline (sentence-transformers + ChromaDB) for document analysis.

AI Chain Detail

CHAIN 1 — SUMMARIZE

SummarizationChain

<4000 tokens: Stuffing strategy — single Claude call with full text.
≥4000 tokens: Map-Reduce — chunk→summarise each→reduce to final.
Token estimate: 1 token ≈ 4 chars.

CHAIN 2 — EXTRACT

EntityExtractionChain

Extracts structured JSON: names · dates · clauses · tasks · risks.
Retries up to 3× on malformed JSON. Schema-validated via jsonschema.
Markdown code-fence stripping built in.

CHAIN 3 — WORKFLOW

WorkflowGenerationChain

2-step: Extract objectives → Generate numbered steps.
Output: JSON array with step_number · action · priority (High/Medium/Low).
Sequential numbering normalised automatically.

AI Testing Principle — Never Call Real LLM in CI Fast Path
All three chains accept an injected llm parameter. Unit and integration tests pass AsyncMock with pre-recorded JSON fixtures. Real Claude is only called in nightly security tests and weekly golden-path E2E tests.

8. Event Bus & Messaging

Domain events travel over RabbitMQ using a TOPIC exchange named rambot.events. All messages are durable. The IEventPublisher interface abstracts the broker — in tests, InMemoryEventPublisher captures events for assertion without a real broker.

10. Design Principles

PrincipleApplied WhereEvidence in Code
Clean ArchitectureAll backend servicesdomain/ → application/ → infrastructure/ → presentation/ — no upward imports
Domain-Driven Designsteer-service, skill-serviceSteerGoal entity (pure dataclass), AlignmentScore value object (frozen), DomainEvents
CQRSsteer-service, skill-serviceSeparate commands/ and queries/ folders in application layer
Dependency Inversion (SOLID-D)AI chains, servicesSummarizationChain(llm=injected), DocumentService(supabase=injected)
Single Responsibility (SOLID-S)All modulesfile_validator.py validates only, security.py handles only JWT, each chain does one thing
Repository Patternsteer-service, skill-serviceAbstract IRepository interface, concrete repo_impl — domain never touches SQLAlchemy directly
Feature-Sliced Designapps/websrc/features/steer/, src/features/skill/ — co-located components + store slices
API Gateway PatternInfrastructureSingle entry point handles cross-cutting concerns (auth, rate limit, logging, compression)
AI-Native Testingdocument-service testsSchema validation over value matching; pre-recorded fixtures; security tests marked separate
Event-Driven Architecturesteer, skill, notificationRabbitMQ TOPIC exchange; IEventPublisher interface; InMemoryPublisher for tests
